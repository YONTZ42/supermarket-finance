# 要件定義書

## 1. 背景
各店舗が独自形式の Excel で決算データを管理しており、店舗横断の比較・集計が困難である。
経営層が半期ごとの売上・経費・利益を共通フォーマットで把握できる仕組みを構築する。

## 2. 目的
- 店舗ごとに異なる管理形式のデータを一元管理する
- 半期（上期・下期）単位で全店舗比較できる
- 初期データとして Excel 由来のデータを表示可能にする
- 将来の店舗追加時に専用ロジック追加を最小化する

## 3. スコープ

### 対象店舗
| 店舗 | 入力方式 | 報告月 |
|---|---|---|
| 東京 | 累積入力 (CUMULATIVE) | 3, 6, 9, 12月 |
| 大阪 | 期間別入力 (PERIODIC) | 3, 6, 9, 12月 |
| 名古屋 | 累積入力 (CUMULATIVE) | 6, 12月 |

### 対象画面
- データ登録画面
- サマリ画面

## 4. 機能要件

### 4.1 データ登録画面
- 店舗選択ができる
- 店舗ごとのカテゴリ（売上/経費）を入力できる
- 店舗ごとの入力タイミング（reportMonths）と入力形式（inputMode）に従って入力できる
- 不正な形式の入力をガードできる（zod バリデーション）
- 登録後、正規化処理と半期サマリ再計算が自動実行される

### 4.2 サマリ画面
- 年度・店舗・半期で一覧表示できる
- 売上合計・経費合計・利益を表示できる
- 店舗横断比較ができる
- グラフで概況を確認できる

### 4.3 初期データ表示
- 配布された Excel を初期データソースとして扱う
- 起動時点で Excel 由来データが表示される
- Excel の取り込みは `prisma/seed.ts` で行う

## 5. 非機能要件
- データ処理の正確性を優先する（Decimal(14,2) で金額管理）
- 店舗追加時に設定追加（`store_reporting_profiles` + `categories`）で対応しやすい構造にする
- 監査用に編集履歴を `audit_logs` に保持する
- UI はレスポンシブを考慮する

## 6. 将来拡張
- LLM による自然言語フィルタ操作（計算はさせず、絞り込み条件生成のみ）
- visx へのグラフライブラリ切り替え（adapter 構成で対応）
- 認証・権限制御
